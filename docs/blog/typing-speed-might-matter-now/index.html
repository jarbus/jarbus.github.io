TYPING SPEED MIGHT MATTER NOW

With tools like Claude Code and Codex, I think typing speed is starting to matter more, and I haven’t seen anyone mention this yet.

Before AI, typing speed didn’t matter much for programming, because everyone needed to take time to think through problems, come up with a good solution, and debug. These activities were not typically bound by typing speed. A fast typer might code up a solution faster than a slow typer, but if the slow typer had a better thought process, they could still implement working code in a shorter amount of time.

AI changes the playing field: As assistants become responsible for a larger portion of the thought process, a fast typer who can go back-and-forth with an AI assistant can now think, iterate, and debug much faster than a slow typer, even if the slow typer takes more time to think things through. In other words, more messages/richer prompts == more “thinking”, not less. This can mean two things:

  1. More back-and-forth with an AI over a given duration, allowing a programmer to iterate over a solution faster.
  2. More informative prompts for the same amount of messages over a given duration, improving the average quality of a response.

The benefits are also amplified if you can ping-pong between multiple agents on multiple problems at once, so your typing bandwidth (and thus “reasoning” bandwidth) is fully utilized.

I suspect it’s hard to measure improvements from typing speed quantitatively, but qualitatively, speed feels more impactful now than ever.

For ultra-high-level problems like system design, I doubt typing speed matters at all, since those problems are best tackled with pen-and-paper in my opinion. For the day-to-day low-level work of actually implementing said system, I think it’s starting to matter more and more.
